## 🔧 Setup e Ambiente

### Ambiente Virtual (venv)

**Problema:** O ambiente CI/CD do Claude Code pode não ter venv configurado automaticamente.

**Solução:**
```bash
# Sempre verificar se venv existe antes de rodar comandos
ls -la venv/bin/activate 2>/dev/null || python -m venv venv

# Ativar venv antes de qualquer comando Python
source venv/bin/activate  # Linux/Mac
.\venv\Scripts\Activate.ps1  # Windows
```

**Nota:** README já documenta isso, mas em ambientes CI é fácil esquecer.

## 🧪 Testes

### Testes de Integração e Variáveis de Ambiente

**Problema:** Testes de integração falham com "ANTHROPIC_API_KEY não encontrada" mesmo quando .env existe.

**Causa:** Pytest não carrega `.env` automaticamente. É preciso chamar `load_dotenv()` explicitamente no arquivo de teste.

**Solução:**
```python
import os
from pathlib import Path
from dotenv import load_dotenv

# Carregar .env na raiz do projeto
project_root = Path(__file__).parent.parent.parent  # Ajustar conforme localização
env_path = project_root / ".env"
load_dotenv(dotenv_path=env_path)

# Agora os.getenv("ANTHROPIC_API_KEY") funciona
assert os.getenv("ANTHROPIC_API_KEY"), "API key não encontrada"
```

**Referência:** `tests/integration/test_methodologist_smoke.py` (corrigido em 10/11/2025)

### Pytest Markers

**Problema:** Warnings sobre marcadores desconhecidos (`@pytest.mark.integration`).

**Solução:** Registrar marcadores customizados em `pytest.ini`:
```ini
[pytest]
markers =
    integration: Integration tests that require API access
    unit: Unit tests that use mocks
```

**Arquivo:** `pytest.ini` (criado em 10/11/2025)

## 🤖 LangGraph e Interrupts

### Handling de Interrupts (ask_user)

**Padrão:** Usar `Command(resume=...)` para retomar após interrupt.

**Exemplo:**
```python
from langgraph.types import Command

# Primeira invocação
graph.invoke(state, config=config)

# Loop para processar interrupts
while True:
    snapshot = graph.get_state(config)

    if not snapshot.next:
        break  # Grafo finalizou

    # Processar interrupts
    for task in snapshot.tasks:
        if task.interrupts:
            for interrupt in task.interrupts:
                question = interrupt.value
                answer = input(f"Responda: {question}")

                # Retomar execução com resposta
                graph.invoke(Command(resume=answer), config=config)
```

**Referencia:**
- `core/tools/cli/chat.py:157-159` - CLI interativo
- `tests/integration/test_methodologist_smoke.py:157-160` - Teste automatizado

**Por quê:** Facilita entender e testar módulos sem executar toda a aplicação.

**Padrão:**
```python
"""
Script de validação manual para [módulo].

Valida que [módulo] foi implementado com:

- Feature 2

        validate_module()
        print("✅ TODOS OS TESTES PASSARAM!")
    except AssertionError as e:
```

## 🚨 Armadilhas Comuns

### 1. Esquecer de carregar .env em testes

**Sintoma:** `os.getenv("ANTHROPIC_API_KEY")` retorna `None` mesmo com .env configurado.

**Solução:** Sempre adicionar no início do arquivo de teste:
```python
from dotenv import load_dotenv
load_dotenv()
```

### 2. Windows vs Linux: Comandos de Shell

**Problema:** Comandos em docs/process/implementation/ podem usar sintaxe Linux.

**Solução:**
- Documentar versões PowerShell E Bash
- Preferir comandos Python multiplataforma quando possível

**Exemplo:**
```bash
# ❌ Linux-only
export PYTHONPATH=.

# ✅ Python (funciona em todos OS)
import sys
sys.path.insert(0, str(project_root))
```

### 3. pytest.ini vs pyproject.toml

**Escolha:** `pytest.ini` foi escolhido por ser mais simples e focado.

**Alternativa:** `pyproject.toml` seria melhor para projetos com mais ferramentas (black, mypy, etc).

### 4. Confundir Feedback de Hook com Comandos do Usuário

**Problema:** Git hooks (como `stop-hook-git-check.sh`) emitem mensagens como "There are uncommitted changes", mas isso NÃO é um comando do usuário.

**Armadilha:** Claude Code pode interpretar feedback de hook como instrução explícita e agir sem confirmação (ex: commitar automaticamente sem o usuário pedir).

**Solução:**
- ✅ Hook feedback é apenas **informação de contexto**
- ✅ Só agir (commit, push, etc) quando usuário **explicitamente pedir**
- ✅ Se incerto, perguntar ao usuário antes de executar ações destrutivas

**Exemplo correto:**
```
Hook: "There are uncommitted changes"
Claude: [Não age automaticamente]
Usuário: "por favor, commite isso"
Claude: [Agora sim, commita]
```

### 5. Criar Documentação Sem Avisar

**Problema:** Criar arquivos .md extras (tipo CORREÇÃO_X.md, FIX_Y.md) sem avisar o usuário.

**Armadilha:** Violar princípio de responsabilidade única documentado no próprio .claudecode.md. A correção já está documentada nos commits, não precisa de arquivo separado.

**Solução:**
- ✅ SEMPRE perguntar antes de criar novos arquivos de documentação
- ✅ Verificar se informação já está documentada em outro lugar (commits, issues, etc)
- ✅ Seguir tabela de responsabilidades: README, ROADMAP, ARCHITECTURE, docs/process/implementation/, .claudecode
- ❌ NUNCA criar docs "temporários" ou "de correção" sem permissão explícita

**Exemplo correto:**
```
Claude: [Fez correção no código]
Claude: "Correção commitada com mensagem detalhada. Quer que eu crie um documento explicando?"
Usuário: "não, os commits já documentam"
Claude: [Não cria arquivo extra]
```

## 🔄 Fluxo de Trabalho Típico

### Implementar Nova Funcionalidade

1. **Ler contexto obrigatório:**
   - ROADMAP.md (critérios de aceite)
   - ARCHITECTURE.md (estrutura técnica)
   - docs/process/implementation/ (processo)

2. **Criar todo list:**
   ```python
   TodoWrite([
       {"content": "Tarefa 1", "status": "pending"},
       {"content": "Tarefa 2", "status": "pending"}
   ])
   ```

3. **Implementar com TDD onde aplicável:**
   - Lógica de negócio: TDD obrigatório
   - UI/CLI: Manual OK

4. **Validar antes de commit:**
   - Rodar testes unitários
   - Criar script de validação manual
   - Rodar script de validação

5. **Commit + Push:**
   - Mensagem descritiva
   - Incluir "Task X.Y" no final

6. **Atualizar documentação:**
   - ROADMAP.md: Marcar task como concluída
   - ARCHITECTURE.md: Se mudou estrutura
   - README.md: Se mudou setup/comandos
