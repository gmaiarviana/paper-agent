## 🧪 Testes

### Testes de Integração e Variáveis de Ambiente

**Problema:** Testes de integração falham com "ANTHROPIC_API_KEY não encontrada" mesmo quando .env existe.

**Causa:** Pytest não carrega `.env` automaticamente. É preciso chamar `load_dotenv()` explicitamente no arquivo de teste.

**Solução:**
```python
import os
from pathlib import Path
from dotenv import load_dotenv

# Carregar .env na raiz do projeto
project_root = Path(__file__).parent.parent.parent  # Ajustar conforme localização
env_path = project_root / ".env"
load_dotenv(dotenv_path=env_path)

# Agora os.getenv("ANTHROPIC_API_KEY") funciona
assert os.getenv("ANTHROPIC_API_KEY"), "API key não encontrada"
```

**Referência:** `tests/core/integration/smoke/test_methodologist_smoke.py` (corrigido em 10/11/2025)

### Pytest Markers

**Problema:** Warnings sobre marcadores desconhecidos (`@pytest.mark.integration`).

**Solução:** Registrar marcadores customizados em `pytest.ini`:
```ini
[pytest]
markers =
    integration: Integration tests that require API access
    unit: Unit tests that use mocks
```

**Arquivo:** `pytest.ini` (criado em 10/11/2025)

## 🤖 LangGraph e Interrupts

### Handling de Interrupts (ask_user)

**Padrão:** Usar `Command(resume=...)` para retomar após interrupt.

**Exemplo:**
```python
from langgraph.types import Command

# Primeira invocação
graph.invoke(state, config=config)

# Loop para processar interrupts
while True:
    snapshot = graph.get_state(config)

    if not snapshot.next:
        break  # Grafo finalizou

    # Processar interrupts
    for task in snapshot.tasks:
        if task.interrupts:
            for interrupt in task.interrupts:
                question = interrupt.value
                answer = input(f"Responda: {question}")

                # Retomar execução com resposta
                graph.invoke(Command(resume=answer), config=config)
```

**Referencia:**
- `core/tools/cli/chat.py:157-159` - CLI interativo
- `tests/core/integration/smoke/test_methodologist_smoke.py:157-160` - Teste automatizado

### 2.5. fastapi 0.115 + reflex 0.9 = conflito de starlette

**Sintoma:** `pip install` reclama que `fastapi 0.115.x requires starlette < 0.47.0, but starlette 1.0.0 was installed via reflex`.

**Causa:** reflex 0.9 exige `starlette>=0.47`; fastapi 0.115 exige `starlette<0.47`. fastapi 0.136+ relaxou pra `starlette>=0.46`. O conflito aparece em ambientes que já têm fastapi 0.115 instalado de outro projeto e tentam atualizar reflex.

**Solução:**
- `pip install -U fastapi` (vai pra 0.136+, resolve o conflito), ou
- venv limpo (`python -m venv .venv` + `pip install -r requirements.txt`).

**Não fixar fastapi em `requirements.txt`** — fastapi não é dependência direta nossa. O fix vive no ambiente do dev.

### 2.6. Reflex `rx.State`: dataclasses tipados, não `rx.Base` (plataforma de workflow)

Descoberto na migração Streamlit→Reflex da plataforma (`tools/workflow_platform/`,
W-PILOTO-UX-1). Vale para os épicos irmãos (UX-2/3/4) e qualquer app Reflex do repo.

- **`rx.Base` não existe neste build** (`reflex-base==0.9.0`). Para shapes tipados
  em `rx.State` (necessário para `rx.foreach` aninhado e acesso `item.campo` no
  render), use **`@dataclasses.dataclass`** com defaults em todos os campos. Ver
  `tools/workflow_platform/web/view_models.py`.
- **Nomes de campo não podem colidir com métodos de `ObjectVar`** (`items`, `keys`,
  `values`). Um campo `items` quebra com `TypeError: Unsupported type method for
  LiteralVar ... ObjectVar.entries`. Renomeie (usamos `cards` em vez de `items`,
  `item_type` em vez de `type`).
- **`foreach` só itera tipo concreto.** `rx.foreach` sobre `dict["chave"]` (item
  aninhado, tipo `Any`) levanta `ForeachVarError`. Modele o aninhamento com
  dataclasses tipados (`list[QueueGroup]` com `cards: list[QueueItemView]`).
- **Não concatene item de dict untyped com `str` no render** (`d["x"] + "..."` →
  `unsupported operand ObjectItemOperation + str`). Pré-monte a string no backend
  (Python) e renderize pronta. Campos de state tipados (`str`) concatenam normal.
- **NÃO nomeie subpacote local igual a módulo da stdlib (`queue`, `types`, `io`…).**
  O pacote local `queue/` sombreava a stdlib `queue` quando o dir da app entra em
  `sys.path[0]` (o que `multiprocessing`/`urllib3`/`socketio` importam) → o worker do
  granian morre com `AttributeError: module 'queue' has no attribute 'LifoQueue'` /
  `ImportError: cannot import name 'Empty' from 'queue'`. **Pega diferente por SO:**
  no POSIX o worker é *forkado* (herda `sys.modules`, mascara — passava); no Windows é
  *spawned* (reimporta com o dir no path → crash, backend morre, página serve só o
  shell "loading"). Fix aplicado: o pacote foi renomeado para `job_queue/`
  (W-PILOTO-UX-1) — `import queue` resolve pra stdlib em qualquer plataforma. Suba com
  `reflex run` (console script).
- **`reflex init`/`reflex run` mexe no diretório:** deleta o `__init__.py` do
  app-root e cria `requirements.txt`/`.gitignore` próprios. Restaure o `__init__.py`
  (`git checkout`) e não commite os gerados — o `.gitignore` raiz os ignora
  (`.web/`, `.states/`, `assets/external/`, `tools/workflow_platform/requirements.txt`
  e `tools/workflow_platform/.gitignore`). **Comentário no `.gitignore` só em linha
  própria** — `path  # x` vira padrão literal e não casa nada.
- **Portas:** a plataforma usa 3001/8001 para coexistir com o Ensaio (3000/8000).

### 4. Confundir Feedback de Hook com Comandos do Usuário

**Problema:** Git hooks (como `stop-hook-git-check.sh`) emitem mensagens como "There are uncommitted changes", mas isso NÃO é um comando do usuário.

**Armadilha:** Claude Code pode interpretar feedback de hook como instrução explícita e agir sem confirmação (ex: commitar automaticamente sem o usuário pedir).

**Solução:**
- ✅ Hook feedback é apenas **informação de contexto**
- ✅ Só agir (commit, push, etc) quando usuário **explicitamente pedir**
- ✅ Se incerto, perguntar ao usuário antes de executar ações destrutivas

**Exemplo correto:**
```
Hook: "There are uncommitted changes"
Claude: [Não age automaticamente]
Usuário: "por favor, commite isso"
Claude: [Agora sim, commita]
```

### 4.5. Edições longas: chunk por seção é o default

**Regra:** o **gatilho é o tamanho do `new_string` projetado** ou a
**independência das regiões tocadas** — não o tamanho do arquivo. Quando
um dos dois dispara, preferir **um `Edit` por seção**. **CLAUDE.md trata
isso como instrução durável** — esta seção é o rationale.

**Heurística operacional:**
- `new_string` projetado > ~80 linhas → quebrar (este é o gatilho
  principal — é a serialização do argumento que trava o stream).
- Edit cobre seções **independentes** do arquivo (separadas por outros
  blocos não-tocados, ou logicamente desacopladas: Objetivo vs. Testes,
  por exemplo) → preferir um Edit por região. Não vale para parágrafos
  consecutivos do mesmo bloco.
- Arquivo novo grande (> ~150 linhas) → `Write` com esqueleto curto +
  `Edit`s preenchendo seção a seção.
- Edit pequeno em arquivo grande, ou várias mudanças contidas num bloco
  contínuo → Edit único, sem fragmentar. Tamanho do arquivo sozinho
  **não** é gatilho.

**Por que isso ganha fluidez (não só evita erro):** cada Edit pequeno
completa em segundos, mantém o stream ativo, e dá pontos naturais de
checkpoint — se algo der errado num bloco, os anteriores já estão
aplicados. Reescrita monolítica trava o stream por dezenas de segundos
serializando o argumento e, quando falha, perde tudo.

**Sintoma quando a regra é violada:** `API Error: Stream idle timeout -
partial response received`. Causa: o backend Anthropic fecha o stream se
ele fica ocioso (sem chunk de output) — um único `Edit` com `new_string`
muito longo demora pra serializar e dispara o timeout. Sessões com 20+
tool calls acumulados ficam mais frágeis e antecipam o limite.

**Quando cair mesmo assim:**
- ✅ A sessão normalmente é preservada — `/resume` ou pedir pra continuar
  costuma funcionar.
- ✅ Em Claude Code 2.1.105+, env var `CLAUDE_STREAM_IDLE_TIMEOUT_MS`
  estende a janela; usar como mitigação **adicional**, não substituto da
  tática de chunks.

**Onde mais pega:** refinamento estratégico, escrita de épicos detalhados,
edições em ROADMAPs longos, criação de specs de skill. Leitura/análise
pura raramente dispara.

**Referência:** descoberta em sessão `claude/optimize-dev-workflow-aiYYj`
(2026-04-29) durante refinamento de PROTO-WORKFLOW-FILA. Promovida a
default proativo em `claude/fix-api-timeout-s5A2M` (2026-04-30) após
recorrência do sintoma.

**Armadilha:** Violar princípio de responsabilidade única documentado no próprio .claudecode.md. A correção já está documentada nos commits, não precisa de arquivo separado.

## 🧩 Padrões Arquiteturais

### Precedência de modelo LLM (`get_agent_model`)

**Descoberta (POC-ENSAIO):** YAMLs de agente (`core/config/agents/*.yaml`) hardcodam `model:`, então trocar o modelo via `.env` não propagava enquanto o YAML tivesse valor. Hoje o `core/agents/memory/config_loader.get_agent_model` prioriza env antes do YAML:

```
LLM_MODEL  >  ANTHROPIC_MODEL  >  YAML.model  >  DEFAULT_MODEL
```

**Consequência prática:** trocar de modelo em toda a aplicação = editar só `LLM_MODEL` no `.env` e reiniciar. Não precisa editar N YAMLs.

**Armadilha:** se rodar smoke test / validação em CI sem `LLM_MODEL` setado, cai no YAML — confirme o modelo efetivo no log do agente.

### Custo defensivo (`CostTracker`)

Modelos não mapeados em `CostTracker.PRICING` retornam custo zero + warning no log em vez de `ValueError`. Métricas de custo são secundárias e não devem derrubar o chat. Ao adicionar modelo novo, incluir pricing em `core/utils/cost_tracker.py` (ver tabela em https://www.anthropic.com/pricing).
